The right way to write checkpoints and restart files in legacy FEniCS

Practical notes for legacy FEniCS users running transient PDE simulations with XDMF/HDF5 checkpoints, especially in multiphysics, ALE, or MPI settings.

Main idea. A restart file should store the discrete state needed to resume the algorithm, not just a few fields that look physically important. A checkpoint is successful only if it lets you reconstruct the next step consistently.

1. What a good restart file should contain

For a time-dependent simulation, use this rule:

  • Save every field that appears in the residual at the next step.
  • Save enough history to reconstruct all discrete time derivatives.
  • Save fields in the same function spaces you intend to read them back into.
Generic examples.
  • Current primary unknowns, such as velocity, pressure, displacement, temperature, concentration, phase field, and electric potential.
  • Previous-step values whenever your scheme uses terms like (u^{n+1}-u^n)/\Delta t.
  • Older history levels if your scheme uses multistep formulas.
  • Any internal variables, prestress fields, material-history variables, or auxiliary fields needed by the constitutive update.

If you save only the current solution, the restart may still run, but it may not reproduce the original discrete state closely.

2. Naming: one of the easiest ways to break restart

In legacy FEniCS, checkpoint I/O is tied to the function name used in write_checkpoint and read_checkpoint. If names are inconsistent, restart becomes fragile.

Best practice

  • Choose stable, human-readable names such as "Velocity", "Pressure", "Displacement", "Temperature", "Concentration".
  • Rename functions explicitly before writing.
  • Read them back using the exact same names.
  • Do not rely on implicit names or whatever happens to be attached to a function object.

Safe naming pattern

u.rename("Displacement", "Displacement")
p.rename("Pressure", "Pressure")
c.rename("Concentration", "Concentration")

Why random names like f_32 happen

This usually happens when one of these is true:

  • The function was never renamed explicitly.
  • The function came from a split, projection, interpolation, or temporary operation and inherited an internal autogenerated name.
  • A helper routine used func.name() without checking whether the current name is meaningful.
  • The name was changed in one script but the reader still expects the old name.
Trap. If you write using func.name() and the function currently has a name like f_32, then your checkpoint file will contain a dataset with that name. Later you may try to read "Pressure" and fail, even though the file itself is not corrupted.

Good helper for renaming

def rename_functions(rename_dict):
    for func, (name, label) in rename_dict.items():
        func.rename(name, label)

rename_functions({
    velocity: ("Velocity", "Velocity"),
    pressure: ("Pressure", "Pressure"),
    state_old: ("StateOld", "StateOld")
})

3. Recommended XDMF checkpoint writing pattern

Use write_checkpoint(..., append=False) for the first field in a new file, then append=True for the rest.

Do not write all fields with append=True on a fresh file.

Do not mix a separate write(mesh) call into the same checkpoint file unless you know exactly why.

Minimal writer

from dolfin import XDMFFile, MPI

with XDMFFile(MPI.comm_world, "checkpoint100.xdmf") as f:
    f.parameters["functions_share_mesh"] = True
    f.parameters["rewrite_function_mesh"] = False
    f.parameters["flush_output"] = True

    f.write_checkpoint(field_a, "FieldA", 100, XDMFFile.Encoding.HDF5, append=False)
    f.write_checkpoint(field_b, "FieldB", 100, XDMFFile.Encoding.HDF5, append=True)
    f.write_checkpoint(field_c, "FieldC", 100, XDMFFile.Encoding.HDF5, append=True)

Preferred restart writer for transient problems

with XDMFFile(MPI.comm_world, f"checkpoint{step}.xdmf") as f:
    f.parameters["functions_share_mesh"] = True
    f.parameters["rewrite_function_mesh"] = False
    f.parameters["flush_output"] = True

    f.write_checkpoint(state,        "State",        step, XDMFFile.Encoding.HDF5, append=False)
    f.write_checkpoint(state_old,    "StateOld",     step, XDMFFile.Encoding.HDF5, append=True)
    f.write_checkpoint(mesh_disp,    "MeshDisp",     step, XDMFFile.Encoding.HDF5, append=True)
    f.write_checkpoint(mesh_disp_old,"MeshDispOld",  step, XDMFFile.Encoding.HDF5, append=True)
    f.write_checkpoint(scalar,       "ScalarField",  step, XDMFFile.Encoding.HDF5, append=True)
    f.write_checkpoint(scalar_old,   "ScalarFieldOld", step, XDMFFile.Encoding.HDF5, append=True)

4. Recommended reading pattern

Read back into functions on the correct spaces, then assign into mixed functions if needed.

field_a_r = Function(space_a)
field_b_r = Function(space_b)

with XDMFFile(MPI.comm_world, "checkpoint100.xdmf") as f:
    f.read_checkpoint(field_a_r, "FieldA")
    f.read_checkpoint(field_b_r, "FieldB")

If a mixed function is used in the solve

assigner = FunctionAssigner(mixed_space, [subspace_a, subspace_b])
assigner.assign(mixed_function, [field_a_r, field_b_r])

After reading, finalize vectors

def sync_function(f):
    f.vector().apply("insert")

sync_function(field_a_r)
sync_function(field_b_r)

5. The right restart mindset

Restart is not only “load the current solution”.

If your residual contains terms like

(u - u_old)/Dt
(mesh_disp - mesh_disp_old)/Dt
(phi_old - phi_old_old)/Dt

then your restart file should contain the history needed to reconstruct them.

Generic example: reconstructing an auxiliary rate from history

rate = Function(V)
rate.vector().zero()
rate.vector().axpy( 1.0/float(Dt), current.vector())
rate.vector().axpy(-1.0/float(Dt), previous.vector())
rate.vector().apply("insert")

This is often more reliable than trusting a separately saved auxiliary field, especially if that field can become stale or inconsistent.

6. Best practice for mixed spaces

Store subfields separately unless you have a very good reason not to.

# Better for restart portability
f.write_checkpoint(component_1, "Component1", step, XDMFFile.Encoding.HDF5, append=False)
f.write_checkpoint(component_2, "Component2", step, XDMFFile.Encoding.HDF5, append=True)

Writing mixed functions directly can work, but subfields are often easier to inspect, test, and reload.

7. Common mistakes

Mistake What happens
Writing every field with append=True in a new file The file structure can be malformed or incomplete.
Calling write(mesh) and then mixing in checkpoint datasets carelessly The XDMF/HDF5 layout may not match what read_checkpoint expects.
Saving only current variables but not old variables The restarted time derivative is wrong at the first resumed step.
Saving an auxiliary rate field but not the history needed to rebuild it You may restart from a field that looks valid but is inconsistent with the discrete update.
Using func.name() without checking the current name The file may end up containing datasets named f_12, f_32, or other autogenerated labels.
Renaming in the writer but not matching the same names in the reader read_checkpoint fails or reads the wrong dataset.
Not creating output directories explicitly Writes fail or behave inconsistently across environments.
Not flushing output Files may look incomplete after interruption.
Trusting checkpoint data without a readback test Restart bugs stay hidden until much later.

8. A simple write-readback test you should always do

After adding or modifying checkpoint logic, test one write and one immediate readback:

# write
with XDMFFile(MPI.comm_world, "debug_checkpoint.xdmf") as f:
    f.parameters["functions_share_mesh"] = True
    f.parameters["rewrite_function_mesh"] = False
    f.parameters["flush_output"] = True
    f.write_checkpoint(field_a, "FieldA", step, XDMFFile.Encoding.HDF5, append=False)
    f.write_checkpoint(field_b, "FieldB", step, XDMFFile.Encoding.HDF5, append=True)

# read back
field_a_chk = Function(space_a)
field_b_chk = Function(space_b)
with XDMFFile(MPI.comm_world, "debug_checkpoint.xdmf") as f:
    f.read_checkpoint(field_a_chk, "FieldA")
    f.read_checkpoint(field_b_chk, "FieldB")

print("||field_a - field_a_chk|| =", norm(field_a.vector() - field_a_chk.vector()))
print("||field_b - field_b_chk|| =", norm(field_b.vector() - field_b_chk.vector()))
Rule. If a write-readback test fails, do not trust restart behavior until the checkpoint path is fixed.

9. What to save for exact restart vs practical restart

Goal Recommended fields
Practical restart that runs robustly Current primary unknowns plus any internal-history variables required by the constitutive model
Restart that reproduces first-step residuals closely Current fields, previous-step fields, and any older levels used by multistep schemes
Debugging ALE or moving-mesh consistency Prefer storing current and previous mesh positions, then derive mesh velocity from them

10. A compact checklist

  • Use explicit variable names like "Velocity", "Pressure", "Temperature", "Concentration".
  • First checkpoint field: append=False. All later fields: append=True.
  • Set flush_output=True.
  • Save history variables, not only current variables.
  • After changing checkpoint logic, run a write-readback test.
  • Do not rely on autogenerated function names.
  • For moving-mesh problems, prefer reconstructing mesh velocity from position history unless the stored velocity has been verified.

11. Bottom line

The right checkpoint strategy in legacy FEniCS is not just about syntax. It is about saving the discrete state your solver actually needs at restart. If the residual depends on history, save the history. If a derived quantity can be reconstructed from more fundamental variables, store the fundamental variables. Use stable names, not accidental ones. And always verify checkpoint fidelity with an immediate write-readback test.
