Checkpoint & Restart in Legacy FEniCS

Category: FEniCS

Last updated: ...

For transient PDE simulations in legacy FEniCS, storing the discrete state accurately is critical for resuming simulations without numerical drift. This involves more than just saving final solution fields.

The Rule of Thumb.

A checkpoint must store everything needed to reconstruct the residual at the next step, including historical fields for time derivatives.

Consistent Naming

In legacy FEniCS, I/O is tied to the function's name() attribute. Always rename your functions explicitly before writing to avoid fragile autogenerated names like f_32.

u.rename("Velocity", "Velocity")
p.rename("Pressure", "Pressure")

Recommended XDMF Writing Pattern

Use write_checkpoint with append=False for the first field and append=True for subsequent fields in the same HDF5/XDMF file.

with XDMFFile(MPI.comm_world, "checkpoint.xdmf") as f:
    f.parameters["functions_share_mesh"] = True
    f.parameters["flush_output"] = True

    f.write_checkpoint(u, "Velocity", step, XDMFFile.Encoding.HDF5, append=False)
    f.write_checkpoint(p, "Pressure", step, XDMFFile.Encoding.HDF5, append=True)
Transient History.

If your scheme uses $u^{n+1}-u^n$, you MUST save $u^n$ in the checkpoint. Saving only $u^{n+1}$ will break the discrete time derivative upon restart.

Common Pitfalls

Action Result
Mixing write(mesh) with checkpoints Corrupts the XDMF layout; read_checkpoint will fail.
Ignoring autogenerated names The reader will look for "Pressure" but fine a dataset named "f_12".
Not flushing output Files may look incomplete if the cluster job crashes.
Verification.

Always perform a write-readback test: write a checkpoint and immediately reload it into new functions. Compare the norms to ensure zero loss of fidelity.

Conclusion

  1. Rename every function explicitly.
  2. Save the history levels needed for the time-stepping algorithm.
  3. Use write_checkpoint consistently across MPI ranks.
