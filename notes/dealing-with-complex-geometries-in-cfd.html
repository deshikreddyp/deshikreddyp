Dealing with complex geometries in CFD

Marking Boundaries

Biomedical geometries are typically constructed from UV or PC MRI scans. PC MRI scans are among the most advanced methods for capturing the three-dimensional details of an area of interest (AOI). Recent developments in 4D MRI techniques have proven to have higher resolution than computational meshing.

PC MRI images are voxelated using available software and then connected through triangles to form a surface mesh. However, STL files resulting from this process cannot be segmented according to our requirements as they only provide information about triangles and their normals, without connectivity and topology information.

To impose boundary conditions, we developed a code that generates face groups based on the angles between neighboring triangles. The algorithm iterates over all facets, checks the angles between them and their neighbors, and assigns them to the appropriate group. Note: The code supports the gmsh .msh V2.2 format.

Conforming Meshes

In computational fluid dynamics (CFD), solving coupled problems often requires a Monolithic approach (strongly coupled). This approach involves the simultaneous solution of equations for both subdomains, ensuring high accuracy at the interface.

The Monolithic approach requires mesh conformity—continuous elements where nodes on both sides of the interface match exactly. This is particularly critical for fluid-structure interaction (FSI) problems consisting of fluid and solid subdomains.

💡 GMSH Workflow: For high-fidelity simulations, GMSH is favored for its flexibility. Since most MRI-derived sources are .STL files, the workflow involves:

  1. Convert .STL files into .STEP or .brep using CAD software or GMSH (OpenCascade).
  2. Read .STEP files in GMSH and perform boolean operations.
  3. Use boolean fragments or coherence operations to generate a conformal geometry.
  4. Mesh the geometry and specify physical regions for each subdomain.

Challenges in Conformal Meshing

  • Format Limitations: Native STL files lack the topological data needed for conformal operations in GMSH. OpenCascade formats like .STEP or .brep are required.
  • Normal Inversion: Reading .STEP files in GMSH can occasionally invert facet normals randomly. This is addressed in the mark_boundaries.py utility.

GMSH → FEniCS Workflow

While dolfin-convert supports meshes from GMSH V2.2, for recent GMSH versions, meshio is the recommended package. Converting to .h5 (HDF5) format is generally best for FEniCS/FEniCSx workflows.

import os
import sys
import dolfin as df
from mark_boundaries import *

def msh_to_h5():
    """Convert given geo-file to a h5-mesh."""
    msh_input_file = sys.argv[1]
    h5_output_file = sys.argv[2]
    # Convert msh-mesh to xml-mesh
    os.system("dolfin-convert {0}.msh {0}.xml".format(msh_input_file))

    # Read xml-mesh
    mesh = df.Mesh("{}.xml".format(msh_input_file))

    # Mark domains (volumetric subdomains)
    domains = df.MeshFunction("size_t", mesh, "{}_physical_region.xml".format(msh_input_file))
    df.File("domains.pvd") << domains

⚠️ Common Errors:

KeyError during conversion: Usually caused by trying to convert a physical region of incorrect dimension. Ensure your physical tags match the mesh dimensionality.
