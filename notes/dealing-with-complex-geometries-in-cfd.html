Dealing with complex geometries in CFD

Marking boundaries:

Biomedical geometries are typically constructed from UV or PC MRI scans. PC MRI scans are among the most advanced methods for capturing the three-dimensional details of an area of interest (AOI). Recent developments in 4D MRI techniques have proven to have higher resolution than computational meshing.

PC MRI images are voxelated using available software and then connected through triangles to form a surface mesh, which is the best possible outcome of MRI images. However, STL files resulting from this process cannot be segmented according to our requirements and only provide information about triangles and their normals, without connectivity and topology information.

To impose boundary conditions, we developed a code that generates face groups based on the angles between neighboring triangles. The algorithm iterates over all facets, checks the angles between them and their neighbors, and assigns them to the appropriate group. Please note that the code only supports the gmsh .msh V2.2 format.

Conforming Meshes:

In the field of computational fluid dynamics (CFD), solving coupled problems can be challenging. One way to solve coupled problems is through a strongly or weakly coupled approach. The strongly coupled approach involves the simultaneous solution of equations for both subdomains. One of the most common strongly coupled approaches is the 'Monolithic' approach.

The Monolithic approach requires mesh conformity near the interface. This means that the mesh near the interface should be continuous, and the nodes on both sides of the interface should match. This ensures that the solution is accurate, and the interface between the subdomains is well-defined.

In our case, we have a fluid-structure interaction (FSI) problem, which consists of two subdomains - fluid and solid - in the mesh. The Monolithic approach is considered the most accurate method for strongly coupled problems, and it is well-suited to solve our FSI problem.

An example of a conformal mesh is shown below. As you can see, the mesh is continuous near the interface, and the nodes on both sides of the interface match. This ensures that the solution is accurate and the interface between the subdomains is well-defined.

Conformal meshes can be generated using software such as ANSYS, GMSH, and Salome. For high-fidelity simulations, GMSH is the go-to software due to its flexible settings. Since most of the geometry source files obtained through MRI scans are in .STL format, a framework for working with STL files is shown below:

  1. Convert .STL files into .STEP files using CAD software or GMSH.
  1. Read the .STEP files in GMSH and perform boolean operations.
  1. Use boolean fragments or coherence operations to fragment the geometry files, which generates a conformal geometry.
  1. Mesh the conformal geometry and specify physical regions as subdomains (e.g., fluid or solid for FSI problems).

Challenges faced to make conformal mesh:

  1. STL files cannot be processed to make a conformal mesh. GMSH requires more data to make the geometry conformal, and is done through OpenCascade supported formats. (.STEP, .brep)
  1. .STEP files when read in GMSH, inverts some facet normals (randomly). This is addressed in mark_boundaries.py file.

GMSH→FEniCS

Dolfin-convert only supports meshes generated from GMSH Version 2.2. You should use the meshio package for the latest GMSH meshes.

Use the following code to convert a GMSH mesh to the .h5 file format, which FEniCS best supports.

To convert the mesh, you must use the dolfin-convert mesh tool, which requires the dolfin package. We also developed the "mark_boundaries.py" code to mark boundaries based on the angle that a facet makes with its neighbors.

"""
Converter from msh-format to a mesh in h5-format.

Usage:
    Usage: python3 <path_to_mshToH5.py> <msh_file> <h5_file>
    E.g.:  python3 mshToH5.py lid lid5
"""

import os
import sys
import dolfin as df
from mark_boundaries import *

The Dolfin .XML format provides flexibility for manipulating mesh data such as facets, normals, edges, and topology.

The dolfin-convert tool converts gmsh .msh files to Dolfin's native .xml format.

def msh_to_h5():
    "Convert given geo-file to a h5-mesh."

    msh_input_file = sys.argv[1]
    h5_output_file = sys.argv[2]
    # Convert msh-mesh to xml-mesh
    os.system("dolfin-convert {0}.msh {0}.xml".format(msh_input_file))

Read the physical regions from the mesh file. The physical regions in this context are volumetric subdomains. Check “domains.pvd” file to check the subdomains in the mesh.

    # Read xml-mesh
    mesh = df.Mesh("{}.xml".format(msh_input_file))

    # Mark domains
    domains = df.MeshFunction("size_t", mesh, "{}_physical_region.xml".format(msh_input_file))
		File("domains.pvd") << domains

We list out the number of domains and tags for few reasons, that will be discussed in the next parts of code

mark_boundaries( ) calls the mark_boundaries.py file and the function which marks the boundaries. The second output of the function ‘tag’ is the maximum tag/marker used in defining the boundaries

    domains_list = np.unique(domains.array())

    # Mark boundaries
    boundaries, tag = mark_boundaries(mesh,35,0,10)
    # mark_boundaries(mesh, threshold_angle=30, I=0, min_facets=10) 
		# (optional arguemnts: I, min_facets)

Now, we check for multi-domain meshes, and differentiate any face which contains two domains.

    if (len(domains_list) > 1): # in case of multi-domain meshes ... 
    # Mark different tags for face with tag '3' by differentiating between domains '1' and '2'
    for f in df.facets(mesh):
        if boundaries[f] == 1:
            if domains[f.entities(3)[0]] == domains_list[0]:
                boundaries[f] = 43
            elif domains[f.entities(3)[0]] == domains_list[1]:
                boundaries[f] = 44

We detect a surface which contains both the domains. This newly detected boundary is the interface between the domains.

    # iterate all the facets of entire mesh
    # check for cells that contains this facet. (each facet is shared by two cells)
    # if any facet is associated with two domains, then it belongs to interface
    for f in df.facets(mesh):
        # check if facet is shared by two cells or not
        if len(f.entities(3)) == 2:
            if domains[f.entities(3)[0]] != domains[f.entities(3)[1]]:
                boundaries[f] = 45    
    File("boundaries.pvd") << boundaries

Write the mesh file in HDF5 format (FEniCS favorite), enclosing domain, boundary data into it.

    # Delete xml-mesh
    os.remove("{}.xml".format(msh_input_file))
    os.remove("{}_physical_region.xml".format(msh_input_file))

    # Write h5-mesh, with domains, boundaries details
    file = df.HDF5File(mesh.mpi_comm(), "{}.h5".format(h5_output_file), "w")
    file.write(mesh, "/mesh")
    file.write(boundaries, "/boundaries")
    file.write(domains, "/domains")

if __name__ == '__main__':
    msh_to_h5()

Possible Errors:

Some of the possible errors you might encounter when marking boundaries or dolfin-convert

  1. Error:

    index = nodes_as_facets[tuple(nodes)]

    KeyError: (2, 32059, 264666)
    Solution:
    You might be trying to convert physical region that is not of the correct dimension. For example, you may be working with 3D volumetric meshes, and tried to convert a physical region that is a surface.
    In a case you want to convert the surface boundaries through th edolfin-convert command, you might have to consider the basic approach of reading the boundaries (the legacy method).
